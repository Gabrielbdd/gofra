## Design Principles

**Explicit over implicit.** Dependencies are passed through struct fields and
function parameters, never accessed via globals. If you read `main.go`, you
can trace every dependency. This is how Go programs are written.

**Protocols over frameworks.** The API contract is Protocol Buffers. The RPC
layer is Connect RPC. The durable execution layer is the Restate SDK. The
frontend data layer is generated from proto. Gofra provides project structure
and development tooling around these protocols — it does not wrap them in its
own abstractions.

**Single source of truth for types.** A `.proto` file defines a resource once.
From it, code generation produces: Go server stubs, Go message types, TypeScript
message types, TypeScript query hooks, and validation rules. No type is
manually written in two places.

**Crash-proof background operations by default.** Any operation that leaves the
HTTP request-response cycle (sending email, processing a payment, indexing
search) runs through Restate's durable execution. If the process crashes, the
operation resumes from its last completed step, not from scratch.

**Ship as a single artifact.** The production build is one Go binary containing
the API server, the Restate service endpoint, and the compiled frontend assets.
Deploy it alongside a Restate Server and a Postgres database. Nothing else.

## 3. Technology Choices and Rationale

### Go

**Reason**: Compiled to a single binary (no runtime to install on servers).
Strong static typing catches errors at compile time. Goroutines handle
concurrency without threading complexity. The standard library covers HTTP,
JSON, crypto, and testing well enough that most applications need few external
dependencies. The language is deliberately simple — there is usually one obvious
way to do things, which matters for a framework that will be read by many
developers.

### Connect RPC (with Protocol Buffers, generated by buf)

**Reason**: REST APIs require hand-writing route definitions, request/response
types, validation, and documentation — in both the server language and every
client language. Connect RPC eliminates this: you define the API once in a
`.proto` file and generate typed server stubs and typed clients.

Connect was chosen over raw gRPC because: Connect handlers are standard
`net/http.Handler` (they compose with any Go HTTP middleware). Connect works in
browsers without a proxy (grpc-web requires Envoy). Connect supports JSON
natively (you can `curl` any endpoint). Connect is wire-compatible with gRPC,
so gRPC clients work too.

buf was chosen over protoc because: buf manages proto dependencies (like
`buf/validate`), lints proto files against style rules, detects breaking
changes between versions, and generates code for multiple languages in a
single command. It replaces the protoc + manual plugin management workflow.

### Google AIP Conventions (for proto design)

**Reason**: Designing protobuf APIs from scratch leads to inconsistent naming,
ad-hoc pagination, and bespoke error handling. Google's API Improvement
Proposals (AIPs) codify design patterns learned from maintaining thousands of
APIs. Adopting the relevant subset gives us consistent conventions enforced by
linting, not by discipline.

We adopt AIPs selectively. The full rationale per AIP follows in Section 8.

### Restate

**Reason**: Traditional web frameworks use database-backed queues for background
work. When a job crashes halfway through, it restarts from the beginning. The
developer must manually handle idempotency, duplicate detection, and partial
failure recovery. This is workable for simple jobs (send an email) but dangerous
for multi-step operations (charge a customer, reserve inventory, create an order).

Restate is a durable execution engine. It journals every step of a handler's
execution. If the handler crashes after step 3, it resumes at step 4 on retry —
steps 1-3 are replayed from the journal without re-executing their side effects.
This makes crash recovery automatic instead of manual.

Restate also provides: Virtual Objects (keyed stateful entities with single-
writer concurrency — no database locks), Workflows (long-running processes that
can suspend waiting for external input), durable timers (sleep for days, survive
restarts), and exactly-once messaging between services.

Restate runs as a single Rust binary. It replaces: a queue system (Sidekiq,
Horizon, Solid Queue), a retry/backoff library, a saga coordinator, a cron
daemon, an event bus, and a workflow engine — with one process.

### PostgreSQL

**Reason**: The source of truth for domain data. Users, posts, orders — the
things the business cares about. Postgres was chosen because: it is mature and
reliable, its full-text search is good enough for most applications (eliminating
a separate search service), it supports JSON columns for semi-structured data,
and it has strong ecosystem support in Go.

Restate has its own embedded K/V store for operational state (shopping cart
contents, workflow progress, notification counters). The two are complementary:
Postgres for data that needs complex queries and joins, Restate K/V for state
tightly coupled to execution logic that benefits from single-writer consistency.

### Frontend: React + Vite + TanStack + shadcn + Tailwind 4

**Reason**: The framework does not do server-side rendering. The Go process
serves an API and static files. The frontend is a standalone SPA.

React was chosen because it has the largest ecosystem. TanStack Router provides
type-safe file-based routing. TanStack Query manages server state (caching,
refetching, optimistic updates). Connect-Query bridges TanStack Query with our
proto-generated types, so every API call is type-safe from proto definition to
React component with zero manual type definitions.

shadcn/ui provides composable, accessible UI components that are copied into the
project (not installed as a dependency), so they can be customized freely.
Tailwind 4 provides utility-first CSS with the new engine that requires no
configuration file.

Vite was chosen because it provides sub-second hot module replacement in
development and produces optimized bundles for production. In development, the
Go server proxies non-API requests to Vite. In production, `web/dist/` is
embedded in the Go binary via `//go:embed`.

**Why the frontend is a separate concern**: The contract between frontend and
backend is the `.proto` file. The frontend can be replaced with Next.js, Svelte,
or a mobile app without changing a single line of Go. The framework scaffolds
the default SPA but does not couple to it.

### mise

**Reason**: Development requires multiple tools at specific versions: Go, Node,
buf, protoc-gen-go, protoc-gen-connect-go, protoc-gen-go-restate, restate-server.
mise pins all tool versions in `mise.toml` so every developer gets the same
environment. It also replaces Makefiles for task running — tasks are defined in
the same file, support dependency ordering, parallel execution, and incremental
builds via source/output tracking.

We use mise instead of a custom `gofra` CLI for tasks because: mise already
exists and is well-maintained, task definitions are declarative TOML (not
compiled Go), and developers can inspect and modify tasks without rebuilding
the CLI.

The `gofra` CLI still exists for code generators (`gofra generate service`,
`gofra generate workflow`) because those need to understand the project structure
and produce Go files with correct imports — something a shell task cannot do well.

```
│  │                        │  │                          │  │
│  │   PostsService         │  │   index.html             │  │
│  │   AuthService          │  │   assets/*.js            │  │

│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │   Restate Service Endpoint                           │  │
│  │   (Restate Go SDK server, :9080)                     │  │
│  │                                                      │  │
│  │   Services:   MailService, SearchIndexer             │  │
│  │   Objects:    NotificationFeed, Scheduler            │  │
│  │   Workflows:  OrderCheckout, PayoutApproval          │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
         │                          │
         │ Restate ingress client   │ invocation push (HTTP/2)
         ▼                          ▼
┌──────────────────────┐    ┌──────────────────────┐
│   Restate Server     │    │     PostgreSQL       │
│   :8080 / :9070 UI   │    │                      │
└──────────────────────┘    └──────────────────────┘
```

One Go binary, two listeners:
- **:3000** — HTTP server (chi mux) serving Connect RPC handlers and embedded
  SPA assets. This is what browsers and API clients talk to.
- **:9080** — Restate service endpoint. This is what the Restate Server pushes
  invocations to. Not exposed to external traffic.

**Reason for one binary, two listeners**: The Connect handlers and Restate
handlers share the same database connection pool and configuration. Running
them in one process avoids inter-process communication and deployment
complexity. The two listeners serve different protocols (standard HTTP/1.1+2
for Connect, HTTP/2 for Restate SDK) to different clients (browsers vs. Restate
Server).

## 5. Project Structure

Two different directory structures matter in Gofra and they must not be mixed
up:

1. The **framework repository** structure, where reusable runtime packages,
   generators, docs, and the canonical starter source live while Gofra itself is being
   built.
2. The **generated application** structure, which is what future
   `gofra new myapp` should produce for application teams.

If these are conflated, the framework starts treating its own repo like a fake
application. That makes generators, starter evolution, and reusable packages harder to
evolve coherently.

### Framework Repository Structure

The framework repo keeps reusable packages at the root and keeps the canonical
generated-app starter under `internal/projectgen/starter/`:

```
gofra/
├── go.mod
├── runtimeconfig/              # Reusable runtime-config resolver + HTTP handler
├── internal/
│   ├── projectgen/
│   │   ├── generator.go        # Embedded starter copy + token replacement
│   │   └── starter/full/       # Canonical generated app source tree
│   └── runtimeconfiggen/       # Generator internals (not public API)
├── cmd/
│   ├── gofra/
│   │   └── main.go             # `gofra new` and future project generators
│   └── gofra-gen-runtimeconfig/
│       └── main.go             # Codegen entrypoint for the runtime-config slice
├── docs/                       # Product contract and architecture docs
├── AGENTS.md
└── CLAUDE.md
```

**Reason for reusable packages at the repo root**: these are the framework
APIs that generated apps should import directly. Hiding them inside starter or
app would invert the dependency direction.

**Reason for `internal/` generator packages**: generator implementation details
should stay private. The public interface is the `gofra` command, not the code
that renders templates.

**Reason for `internal/projectgen/starter/full/`**: `gofra new` needs one
canonical source tree now. Keeping it inside the framework repo makes the
generated app contract explicit, reviewable, and testable without pretending
that the framework repo itself is the application.

**Reason for separating library code from starter-owned files**: reusable
behavior belongs in framework packages such as `runtimeconfig/`. App-specific
wiring, checked-in generated placeholders, and shell files belong in the
starter because they are part of the generated project, not the framework API.

### Current Starter Ownership

The starter is the canonical generated application shape for the current phase,
but it is intentionally smaller than the full target layout below.

- Framework-owned code stays at the repo root and is imported by generated apps
  as a library.
- Starter-owned files live under `internal/projectgen/starter/full/` and are
  copied into new applications by `gofra new`.
- Future post-create generators will write additional app-owned files into the
  generated project after the base starter contract is stable.

Today `gofra new` copies one minimal runnable starter that includes:

- `cmd/app/` for the HTTP entrypoint
- `config/` for typed config plus runtime-config wiring
- `gen/<app>/runtime/v1/` for checked-in placeholder runtime-config Go types
- `proto/<app>/runtime/v1/` for the public runtime-config contract stub
- `web/` for a minimal embedded shell
- `gofra.yaml` and `go.mod`

The generated `go.mod` currently includes a local `replace` directive back to
the framework checkout that created the app. This is a temporary development
contract until the framework module is published.

**Reason for `proto/` at the root**: It is the contract that both `gen/` (Go)
and `web/src/gen/` (TypeScript) are generated from. Placing it at the root
makes its role visible.

**Reason for `app/rpc/` separate from `app/services/`**: RPC handlers implement
Connect interfaces (synchronous, request-response). Services/Objects/Workflows
implement Restate interfaces (durable, async). They have different context
types, different execution models, and different testing strategies. The
directory name tells you which runtime owns the code.

**Reason for `web/` as a standalone project**: It has its own `package.json`,
own build, own test suite. The Go binary doesn't know or care what framework
the frontend uses. The contract is the proto file. The frontend could be
replaced with Svelte, a mobile app, or a third-party integration without
changing any Go code.

**Reason for documenting both layouts in one place**: future work needs one
authoritative answer for where framework code should live and what a generated
application should look like. This section prevents new implementation work from
smearing those boundaries together.

## 6. API Layer: Connect RPC + Protobuf

### Proto File Structure

Every API resource is defined once in a `.proto` file. Following AIP
conventions, each resource gets standard methods with consistent naming:

```protobuf
// proto/myapp/posts/v1/posts.proto
syntax = "proto3";
package myapp.posts.v1;

import "buf/validate/validate.proto";
import "google/protobuf/field_mask.proto";
import "google/protobuf/timestamp.proto";

service PostsService {
  rpc GetPost(GetPostRequest) returns (Post) {}
  rpc ListPosts(ListPostsRequest) returns (ListPostsResponse) {}
  rpc CreatePost(CreatePostRequest) returns (Post) {}
  rpc UpdatePost(UpdatePostRequest) returns (Post) {}
  rpc DeletePost(DeletePostRequest) returns (Post) {}       // soft delete
  rpc UndeletePost(UndeletePostRequest) returns (Post) {}   // custom method
  rpc PublishPost(PublishPostRequest) returns (Post) {}      // custom method
}
```

**Reason for standard method naming (AIP-130–136)**: `GetPost`, `ListPosts`,
`CreatePost` — the verb-noun pattern is predictable. A developer who has seen
one service knows the shape of every service. Generators can scaffold from
conventions instead of configuration. The API linter enforces consistency.

### Resource Message

```protobuf
message Post {
  int64 id = 1;
  string title = 2;
  string slug = 3;
  string body = 4;
  PostStatus status = 5;
  int64 author_id = 6;
  string etag = 7;
  google.protobuf.Timestamp create_time = 8;
  google.protobuf.Timestamp update_time = 9;
  google.protobuf.Timestamp delete_time = 10;
  google.protobuf.Timestamp publish_time = 11;
  User author = 12;
  repeated Tag tags = 13;
}

enum PostStatus {
  POST_STATUS_UNSPECIFIED = 0;
  POST_STATUS_DRAFT = 1;
  POST_STATUS_PUBLISHED = 2;
  POST_STATUS_ARCHIVED = 3;
}
```

**Reason for `google.protobuf.Timestamp` (AIP-142)**: Timestamps are first-class
protobuf types with well-defined semantics. Not `int64` unix seconds, not
`string` ISO-8601. Every language's generated code gets a proper timestamp type.

**Reason for `{action}_time` naming (AIP-148)**: `create_time`, not `created_at`.
`update_time`, not `updated_at`. The `_time` suffix is the AIP standard and
avoids ambiguity with `_at` which some ecosystems interpret differently.

**Reason for `etag` (AIP-154)**: Optimistic concurrency control. Update and
Delete check the etag; if it doesn't match the stored value, the server returns
`Aborted`. This prevents lost updates without database-level locking. The etag
is computed from `update_time` or a content hash.

**Reason for `delete_time` (AIP-164)**: Soft delete. Instead of removing the row,
the handler sets `delete_time`. List excludes soft-deleted resources by default.
Get still returns them. Undelete clears `delete_time`. This matches what Rails
and Laravel call "soft deletes" but follows AIP's field naming.

**Reason for `POST_STATUS_UNSPECIFIED = 0` (AIP-126)**: Protobuf enum zero value
should always be an explicit "unspecified" sentinel, not a real business value.
This prevents bugs where an unset field silently becomes a valid state.

### Request Messages

```protobuf
message ListPostsRequest {
  int32 page_size = 1 [(buf.validate.field).int32 = {gte: 1, lte: 100}];
  string page_token = 2;
  string order_by = 3;
  string filter = 4;
  bool show_deleted = 5;
}

message ListPostsResponse {
  repeated Post posts = 1;
  string next_page_token = 2;
  int32 total_size = 3;
}

message CreatePostRequest {
  string title = 1 [(buf.validate.field).string = {min_len: 1, max_len: 255}];
  string body = 2 [(buf.validate.field).string.min_len = 10];
  repeated string tags = 3;
  string request_id = 4;
  bool validate_only = 5;
}

message UpdatePostRequest {
  Post post = 1 [(buf.validate.field).required = true];
  google.protobuf.FieldMask update_mask = 2;
  bool validate_only = 3;
}

message DeletePostRequest {
  int64 id = 1;
  string etag = 2;
}

message UndeletePostRequest {
  int64 id = 1;
}

message PublishPostRequest {
  int64 id = 1;
  string etag = 2;
}
```

**Reason for pagination from day one (AIP-158)**: Adding `page_size`/`page_token`
to an existing List method is a breaking change — the response shape changes.
Every List method gets pagination fields immediately, even if the table has 10 rows
today.

**Reason for opaque `page_token`** (AIP-158): The server controls the pagination
strategy. The token might encode an offset, a cursor, or a keyset. Clients must
not parse or construct it. This allows the backend to switch from offset to
keyset pagination without breaking clients.

**Reason for `filter` as a string (AIP-160)**: A structured filter field (nested
messages with AND/OR operators) adds proto complexity. A string field with a
simple grammar (`status = "published" AND author_id = 42`) is easier to type
in client code and flexible enough for most use cases. The server parses and
validates it; invalid filters return `InvalidArgument`.

**Reason for `request_id` (AIP-155)**: Client-generated UUID for idempotency. If
the client retries a Create request with the same `request_id`, the server
returns the result of the original request without re-executing. This maps
directly to Restate's idempotency key — the Connect handler passes `request_id`
through to Restate when dispatching durable work.

**Reason for `validate_only` (AIP-163)**: The server validates the request and
returns what the resource would look like, without persisting. Costs nothing to
implement. Enables frontend "dry run" validation before submit.

**Reason for `FieldMask` on Update (AIP-161)**: Protobuf zero values are
indistinguishable from unset fields. Without a field mask, a TypeScript client
sending `{ title: "New Title" }` would cause `body` to be overwritten with
empty string (its zero value). The field mask makes partial updates explicit.

### Validation

**Reason for `buf/validate` over Go-side validation**: Validation rules live in
the proto file, next to the field definition. One source of truth. The
`protovalidate` Connect interceptor applies them automatically — no validation
code in handlers. Both Go and TypeScript codegen can use the same rules (proto
validation is being ported to client-side generation).

### Code Generation

One `buf generate` command produces all artifacts:

```
posts.proto
    │
    ├── protoc-gen-go          → gen/myapp/posts/v1/posts.pb.go
    │                            (Go message types)
    │
    ├── protoc-gen-connect-go  → gen/myapp/posts/v1/postsv1connect/posts.connect.go
    │                            (Go: PostsServiceHandler interface + client)
    │
    ├── protoc-gen-es          → web/src/gen/.../posts_pb.ts
    │                            (TypeScript message types)
    │
    └── protoc-gen-connect-query → web/src/gen/.../posts-PostsService_connectquery.ts
                                   (TypeScript: TanStack Query hooks per RPC)
```

**Reason for generating TanStack Query hooks**: Connect-Query produces `listPosts`,
`getPost`, `createPost` functions that integrate directly with `useQuery` and
`useMutation`. The frontend developer writes `const { data } = useQuery(listPosts, { pageSize: 20 })`
and gets full type safety — the `data` variable has the exact shape of
`ListPostsResponse` as defined in the proto. No manual type definitions. No
`fetch` wrappers. No `any`.

### Error Handling

**Reason for Connect error codes (AIP-193)**: Every Connect error has a typed
code from a fixed set: `InvalidArgument`, `NotFound`, `AlreadyExists`,
`PermissionDenied`, `Unauthenticated`, `Aborted`, `Internal`, `Unavailable`.
This replaces ad-hoc JSON error formats. The frontend can handle errors
generically based on the code. Connect's `ErrorDetail` supports structured
metadata (which field failed validation, what the conflict was).

## 7. Go Server Implementation

### Handler Implementation

Connect RPC handlers implement a generated interface. The `var _ = ...` line
ensures compile-time verification.

```go
type PostsService struct {
    Queries *sqlc.Queries
    Restate *gofra.RestateClient
}

var _ postsv1connect.PostsServiceHandler = (*PostsService)(nil)

func (s *PostsService) ListPosts(
    ctx context.Context,
    req *connect.Request[postsv1.ListPostsRequest],
) (*connect.Response[postsv1.ListPostsResponse], error) {
    pageSize := int32(25)
    if req.Msg.PageSize > 0 {
        pageSize = min(req.Msg.PageSize, 100)
    }

    rows, err := s.Queries.ListPosts(ctx, sqlc.ListPostsParams{
        ShowDeleted: req.Msg.ShowDeleted,
        PageSize:    pageSize + 1,
        PageOffset:  decodePageToken(req.Msg.PageToken),
        Status:      parseStatusFilter(req.Msg.Filter),
    })
    if err != nil {
        return nil, connect.NewError(connect.CodeInternal, err)
    }

    hasNext := len(rows) > int(pageSize)
    if hasNext {
        rows = rows[:pageSize]
    }

    posts := make([]*postsv1.Post, len(rows))
    for i, row := range rows {
        posts[i] = listPostRowToProto(row)
    }

    nextToken := ""
    if hasNext {
        nextToken = encodePageToken(decodePageToken(req.Msg.PageToken) + pageSize)
    }

    return connect.NewResponse(&postsv1.ListPostsResponse{
        Posts:         posts,
        NextPageToken: nextToken,
    }), nil
}
```

**Reason for explicit `*sqlc.Queries` field**: No global database accessor and
no framework-owned query layer. The handler receives typed generated queries
through its struct field, set in `main()`. This is testable and traceable.

**Reason for explicit conversion functions**: sqlc row structs and proto
messages are different types with different concerns. Converting them in
`app/rpc/converters.go` keeps the generated database code untouched and makes
the API mapping explicit.

### HTTP Mux Assembly

```go
func main() {
    cfg, _ := config.Load()
    db, _ := gofra.OpenDB(cfg.Database)
    queries := sqlc.New(db)
    restateClient := gofra.NewRestateClient(cfg.Restate.IngressURL)

    mux := chi.NewRouter()
    mux.Use(middleware.Logger, middleware.Recovery, middleware.RequestID)
    mux.Use(gofra.CORSMiddleware(cfg.CORS))

    interceptors := connect.WithInterceptors(
        gofra.OTELInterceptor(),
        protovalidate.NewInterceptor(),
        gofra.AuthInterceptor(cfg.Auth),
    )

    postsPath, postsHandler := postsv1connect.NewPostsServiceHandler(
        &rpc.PostsService{Queries: queries, Restate: restateClient},
        interceptors,
    )
    mux.Mount(postsPath, postsHandler)

    // SPA: production embeds assets, dev proxies to Vite
    if cfg.App.IsProduction() {
        mux.Handle("/*", gofra.SPAHandler(web.Assets))
    } else {
        mux.Handle("/*", gofra.ViteProxy(cfg.Vite.URL))
    }

    // Restate endpoint
    restateEndpoint := server.NewRestate()
    restateEndpoint.Bind(restate.Reflect(services.MailService{Queries: queries}))
    restateEndpoint.Bind(restate.Reflect(services.SearchIndexer{Queries: queries}))
    restateEndpoint.Bind(restate.Reflect(workflows.OrderCheckout{Queries: queries}))
    restateEndpoint.Bind(restate.Reflect(workflows.PayoutApproval{Queries: queries}))

    gofra.Serve(ctx, gofra.ServeConfig{
        HTTPHandler: mux,
        HTTPAddr:    fmt.Sprintf(":%d", cfg.App.Port),
        RestateAddr: fmt.Sprintf(":%d", cfg.Restate.ServicePort),
        RestateSetup: func() (*server.Restate, error) {
            return restateEndpoint, nil
        },
    })
}
```

**Reason for chi router**: chi implements `net/http` interfaces. Any middleware
that works with `http.Handler` works with chi. Connect RPC handlers return
`(path, http.Handler)` pairs that mount directly on chi. No adapter needed.

**Reason for Connect interceptors (not chi middleware) for validation and auth**:
Connect interceptors receive typed proto requests. A chi middleware receives
raw `http.Request`. Auth and validation operate on the RPC request — they
need the typed message. Using interceptors means the auth logic can inspect
`req.Header()` for tokens and set user context, while validation can check
`req.Msg` fields against `buf/validate` rules.

## 8. Durable Execution Layer: Restate

### What Restate Replaces

| Traditional Subsystem | Restate Replacement | Reason |
|---|---|---|
| Queue + workers (Sidekiq, Horizon) | Restate Services | Every handler IS a durable job. No separate worker process. No queue table. |
| Job retry/backoff config | Restate retry policy | Configured per-service or per-handler. Exponential backoff with configurable bounds. |
| Failed jobs table + management | Restate admin API + UI | Invocation state (pending, running, failed, suspended) is queryable. Retry/cancel via CLI or UI. |
| Event dispatcher + listeners | One-way Restate messages | `ServiceSend` is a durable fire-and-forget message. Delivery is guaranteed. Each listener runs independently. |
| Saga / compensation | `defer` + journaled `Run` blocks | Terminal errors trigger deferred compensations. The journal ensures compensations run even after crashes. |
| Task scheduling / cron | Virtual Object with delayed self-messages | A Scheduler object parses cron expressions, computes next run time, sends a delayed message to itself. Survives restarts. No OS cron. |
| Workflow engine | Restate Workflows | Exactly-once multi-step processes. Can suspend waiting for external signals via Durable Promises. |
| Webhook callbacks | Awakeables / Durable Promises | Handler suspends, external system resolves the promise via HTTP. Handler resumes. |
| Per-entity concurrency control | Virtual Object single-writer | Only one exclusive handler runs per object key at a time. No database locks. No race conditions. |

### What Gofra Still Provides

| Subsystem | Reason it stays |
|---|---|
| sqlc queries + converters | Postgres is the source of truth for domain data. SQL stays explicit. API mapping stays in Go code. |
| Migrations + seeds | Schema management is a database concern. |
| File storage (local/S3) | Orthogonal to execution durability. |
| Cache (memory/Redis) | Response-level caching. HTTP performance concern. |
| HTTP client | For external API calls. Wrapped in `restate.Run()` when inside durable handlers. |
| Full-text search (Postgres FTS) | Query-time concern. Not an execution concern. |
| Authentication + authorization | Identity and permissions are enforced at the request layer. |

### How Handlers Use Restate

Restate handlers are written using the Restate Go SDK directly. Gofra does not
wrap the SDK.

```go
// app/services/mail_service.go
type MailService struct {
    DB     *gofra.DB
    Mailer *mailer.Client
}

func (m MailService) Send(ctx restate.Context, req SendMailRequest) error {
    body, err := restate.Run(ctx, func(ctx restate.RunContext) (string, error) {
        return m.Mailer.RenderTemplate(req.Template, req.Data)
    }, restate.WithName("render"))
    if err != nil {
        return err
    }

    _, err = restate.Run(ctx, func(ctx restate.RunContext) (restate.Void, error) {
        return restate.Void{}, m.Mailer.Send(req.To, req.Subject, body)
    }, restate.WithName("send"))
    return err
}
```

**Reason for using the SDK directly (not wrapping it)**: The Restate SDK is
well-designed. Its context types (`restate.Context`, `restate.ObjectContext`,
`restate.WorkflowContext`) encode capabilities at the type level. Wrapping them
would either lose type safety or duplicate the entire API surface. Developers
learn one SDK, not two.

### Connect → Restate Bridge

Connect handlers dispatch durable work via the Restate ingress client:

```go
func (s *PostsService) CreatePost(
    ctx context.Context,
    req *connect.Request[postsv1.CreatePostRequest],
) (*connect.Response[postsv1.Post], error) {
    // ... create post in DB ...

    // Durable fire-and-forget: guaranteed delivery + guaranteed execution
    if err := s.Restate.Service("SearchIndexer", "Index").Send(
        ctx,
        IndexRequest{PostID: post.ID},
        restate.WithIdempotencyKey(req.Msg.RequestId), // AIP-155 → Restate idempotency
    ); err != nil {
        return nil, connect.NewError(connect.CodeUnavailable, err)
    }

    return connect.NewResponse(postToProto(post)), nil
}
```

**Reason for passing `request_id` as idempotency key**: The client's `request_id`
(AIP-155) can be forwarded to Restate's idempotency system for durable
dispatches. This prevents duplicate Restate sends for the same client request.
The full mutation contract for database writes must still be designed
explicitly; the framework should not imply stronger guarantees than it
implements.

## 9. Event Dispatch

Events are a dispatch table — a map from event names to Restate service
invocations.

```go
var Events = map[string][]Listener{
    "post.created": {
        {Service: "SearchIndexer", Handler: "Index"},
        {Service: "WebhookDelivery", Handler: "Deliver"},
    },
}
```

Dispatch iterates the table and sends one-way Restate messages:

```go
func Dispatch(client *restateingress.Client, event string, payload any, table map[string][]Listener) {
    for _, l := range table[event] {
        restateingress.ServiceSend[any](client, l.Service, l.Handler).
            Send(context.Background(), payload)
    }
}
```

**Reason for a map + loop instead of a pub/sub system**: It is ~15 lines of
code. Every listener invocation is individually durable (Restate guarantees
delivery). Each listener is independently retried on failure. The map is
defined as data, inspectable, and testable. No runtime registration. No
subscriber discovery. No event bus.

## 10. Database Layer

### sqlc Queries

```go
queries := sqlc.New(db)
post, err := queries.GetPostByID(ctx, 42)
rows, err := queries.ListPosts(ctx, sqlc.ListPostsParams{
    ShowDeleted: false,
    PageSize:    25,
    PageOffset:  0,
})
```

**Reason for generated queries over a framework query builder**: The SQL lives
in `db/queries/*.sql`, the schema lives in `db/migrations/*.sql`, and sqlc
generates typed Go methods from both. There is one source of truth and no
framework-owned query DSL to keep coherent.

**Reason for explicit SQL**: complex joins, filters, and pagination rules are a
normal part of application development. SQL is already the language Postgres
executes. Gofra should standardize on typed generated queries rather than hide
SQL behind a custom abstraction.

### Migrations

**Reason for plain SQL files**: SQL is the schema language. A Go DSL adds a
translation layer that obscures what actually happens to the database. Every
migration tool eventually needs `db.Exec(rawSQL)` for the cases the DSL can't
express. Starting with SQL avoids the cliff.

## 11. Frontend

### Development Mode

```
Browser → Go server (:3000)
              │
              ├── Connect RPC handlers
              ├── /_gofra/config.js
              └── Proxies frontend pages/assets/HMR → Vite dev server (:5173)
```

**Reason for Go as the browser entrypoint in dev**: the browser sees one origin
in both development and production. Go can serve API routes and public runtime
config directly, while still proxying frontend requests to Vite for sub-second
hot module replacement.

### Production Mode

```
Browser → Go server (:3000)
              │
              ├── Connect RPC handlers
              ├── /_gofra/config.js
              └── Embedded SPA assets (go:embed web/dist)
```

**Reason for embedding**: Single binary deployment. No separate file server,
no CDN dependency (though one can be placed in front). `embed.FS` serves
files directly from the binary with proper content types and caching headers.

### Browser Runtime Config

The browser gets deploy-time settings from Go, not from baked frontend env
variables. Go serves a browser-safe payload at `/_gofra/config.js`, and
`web/index.html` loads it before the frontend bundle.

The payload is defined by a dedicated runtime-config proto and resolved from
`config.Config` by generated code. Applications can optionally add custom Go
logic for request-aware public values.

This route contains only public values such as OIDC issuer, public client ID,
redirect paths, and API base URL. It never exposes secrets or server-only
configuration.

### SPA Fallback

```go
func SPAHandler(assets embed.FS) http.Handler {
    fsys, _ := fs.Sub(assets, "dist")
    fileServer := http.FileServer(http.FS(fsys))
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if _, err := fs.Stat(fsys, strings.TrimPrefix(r.URL.Path, "/")); err == nil {
            fileServer.ServeHTTP(w, r)
            return
        }
        r.URL.Path = "/"
        fileServer.ServeHTTP(w, r)
    })
}
```

**Reason for fallback to index.html**: TanStack Router handles client-side
routing. A request to `/posts/my-post` should serve `index.html` and let React
render the route. Only actual files (`.js`, `.css`, images) are served directly.

need to be compiled before running any task. mise is a standalone tool that
reads a TOML file. Developers can inspect and modify tasks without touching Go
code.

### Connect Handler Tests

```go
func TestListPosts(t *testing.T) {
    db := gofra.TestDB(t)

```

**Reason for `httptest.Server` + generated client**: This tests the full
request-response path (serialization, validation, handler logic, response
shape) using Connect's own client against a real HTTP server — in-process, no
Docker, millisecond execution.

**Reason for `RestateRecorder`**: In handler tests, we verify that the right
Restate messages were dispatched, without actually running Restate. The
recorder captures `ServiceSend` / `ObjectSend` calls for assertions.

### Restate Handler Tests

```go
func TestMailService(t *testing.T) {
    db := gofra.TestDB(t)
    svc := services.MailService{Queries: sqlc.New(db), Mailer: mailertest.New()}

```

**Reason for Docker-based Restate tests**: Restate handlers depend on the

### Unit Testing Handler Logic

```go
func TestRenderTemplate(t *testing.T) {
```

**Reason for extracting logic into testable functions**: The code inside
`restate.Run()` closures is often the interesting business logic. Extract it

## 14. Production Deployment

### Build

```bash
mise run build
# Produces: gofra-app (single binary, ~30MB)
# Contains: Connect RPC server, Restate endpoint, embedded SPA assets
```

### Runtime Requirements

- The gofra-app binary
- Restate Server binary (or HA cluster)
- PostgreSQL

**No Redis. No message broker. No worker processes. No cron daemon.**

### Startup

1. Binary starts, connects to Postgres
2. HTTP server listens on `:3000`
3. Restate endpoint listens on `:9080`
4. App registers endpoint with Restate Server
5. Scheduled tasks are initialized (idempotent sends to Scheduler object)

### Shutdown (SIGTERM)

1. HTTP server stops accepting new connections
2. In-flight HTTP requests get a deadline to complete
3. Restate endpoint stops accepting new invocations (in-flight invocations are
   retried on another instance by Restate)
4. DB connections close
5. Process exits

**Reason for this shutdown order**: HTTP clients should get responses for
in-flight requests. Restate handles durability of in-progress invocations — it
will retry them on the next available instance. No data is lost.

## 15. Decision Log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Go | Compiled binary. Static types. Goroutines. Standard library. |
| 2 | Connect RPC | Type-safe API from proto. Works in browsers. Standard `net/http`. |
| 3 | buf for codegen | Manages proto deps, lints, detects breaking changes. Replaces protoc. |
| 4 | AIP conventions | Consistent naming, pagination, errors. Enforced by linting. |
| 5 | `{action}_time` fields | AIP-148. `create_time` not `created_at`. One convention, no ambiguity. |
| 6 | Pagination on all List methods | AIP-158. Adding later is breaking. `page_token` is opaque. |
| 7 | `request_id` for idempotency | AIP-155. Client-supplied idempotency key. Forwarded to durable work where applicable. |
| 8 | `FieldMask` on Update | AIP-161. Prevents accidental overwrite with proto zero values. |
| 9 | `validate_only` on mutations | AIP-163. Free to implement. Enables frontend dry-run. |
| 10 | Soft delete with `delete_time` | AIP-164. Consistent with other `_time` fields. Undelete via custom method. |
| 11 | `etag` for concurrency | AIP-154. Optimistic locking without DB-level locks. |
| 12 | Connect error codes | AIP-193. Fixed set of codes. Generic frontend error handling. |
| 13 | Restate for durable ops | Crash recovery at step level. Replaces queue + scheduler + saga + workflow. |
| 14 | Restate SDK direct | No wrapper. One API to learn. Type-safe context hierarchy. |
| 15 | Virtual Objects | Per-key single-writer. No DB locks for shopping carts, notification counts. |
| 16 | Workflows for long-running business flows | Checkout, approvals, external callbacks. Suspend on Durable Promise. |
| 17 | Postgres for domain data | Complex queries, joins, FTS. Restate K/V for operational state. |
| 18 | Explicit DI via struct fields | No service container. No globals. Read `main.go` to trace everything. |
| 19 | Explicit conversion functions between DB rows and API messages | Database structs and proto messages serve different concerns. Keep the mapping explicit. |
| 20 | chi router | `net/http` compatible. Connect handlers mount directly. |
| 21 | SQL migrations | SQL is the schema language. No DSL translation layer. |
| 22 | React + TanStack + shadcn | Largest ecosystem. Connect-Query gives E2E type safety from proto. |
| 23 | Vite | Sub-second HMR. Proxy in dev, embed in prod. |
| 24 | SPA (no SSR) | Decouples frontend and backend. Contract is the proto file. |
| 25 | `embed.FS` for production | Single binary deployment. No separate file server. |
| 26 | mise for tools + tasks | Pins tool versions. Replaces Makefile. Incremental builds. |
| 27 | gofra CLI for project bootstrap and generators | `gofra new` and generators both need Go-aware project structure logic. Tasks stay declarative in TOML. |
| 28 | Events as map + loop | ~15 lines. Each listener is independently durable via Restate. |
| 29 | No server-side rendering or templates | API-first. Frontend is replaceable. |
| 30 | `RestateRecorder` for tests | Fast HTTP tests without Docker. Verify dispatch, not execution. |
| 31 | Docker-based Restate tests | Durable handlers need real journal to test correctly. Tagged `integration`. |
| 32 | Skip AIP-122 (resource names) | Hierarchical names are for infrastructure APIs. Web apps use `id`/`slug`. |
| 33 | Skip AIP-151 (LRO) | Restate Workflows are strictly more capable. Polling-based LRO is a step backward. |
| 34 | Skip AIP-127 (transcoding) | Connect handles HTTP mapping automatically. No annotations needed. |
| 134 | Framework repo uses reusable root packages plus a canonical embedded starter | Keeps framework code distinct from generated app code while giving `gofra new` one testable source of truth. |
