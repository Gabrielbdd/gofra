# Generated App Layout

> Structure and conventions of an application created by `gofra new`.

## Status

Alpha — the generated layout may change before v1.

## Overview

Running `gofra new <directory>` produces a complete, runnable Go application.
This page documents the directory structure, key files, and runtime behavior
of that generated application.

## Directory Structure

```
<app>/
├── cmd/
│   └── app/
│       └── main.go           # Application entrypoint
├── config/                   # Generated after running mise run generate
│   ├── config_gen.go         # Config structs and DefaultConfig()
│   ├── load_gen.go           # NewFlagSet() and Load()
│   └── public_gen.go         # BindPublicConfig() and PublicConfigHandler()
├── db/
│   ├── embed.go              # Embeds migrations into the binary
│   ├── migrations/           # goose SQL migration files
│   │   └── 00001_create_posts.sql
│   ├── queries/              # sqlc query files
│   │   └── posts.sql
│   ├── seeds/                # Seed data (goose --no-versioning)
│   │   └── seed.sql
│   └── sqlc/                 # Generated by sqlc (do not edit)
├── proto/
│   └── <package>/
│       └── config/
│           └── v1/
│               └── config.proto  # Configuration schema (protobuf)
├── web/
│   ├── embed.go              # Embeds web assets into the binary
│   └── index.html            # SPA starter page
├── go.mod                    # Go module with local framework replace
├── gofra.yaml                # Default runtime configuration
├── mise.toml                 # Task runner definitions
├── sqlc.yaml                 # sqlc code generation configuration
├── .gitignore
└── README.md
```

The `config/` directory does not exist immediately after `gofra new`. It is
generated by `mise run generate` (which is a dependency of `mise run dev`,
so it runs automatically on first `dev`). The `db/sqlc/` directory is
generated by `mise run gen:sql` (requires sqlc to be installed).

## Key Files

### `cmd/app/main.go`

The application entrypoint. It:

1. Loads configuration via `config.Load(os.Args[1:])` (the generated
   wrapper, not `runtimeconfig.Load` directly).
2. Opens a PostgreSQL connection pool via `runtimedatabase.Open()`, mapping
   `cfg.Database` fields into `runtimedatabase.Config`. Duration fields
   (e.g., `max_conn_lifetime`) are parsed from Go duration strings.
3. Optionally runs embedded migrations via `runtimedatabase.Migrate()` when
   `cfg.Database.AutoMigrate` is true. Applied migrations are logged.
4. Creates a health checker with `runtimehealth.New()` including a Postgres
   readiness check via `runtimedatabase.HealthCheck(pool)`.
5. Sets up two handler layers:
   - **Root mux** (`http.NewServeMux`) — health probe handlers, structurally
     outside app middleware.
   - **App router** (`chi.NewRouter`) — config endpoint and web assets,
     mounted at `/` on the root mux.
6. Starts the server with `runtimeserve.Serve()`. The database pool is closed
   in the `OnShutdown` callback during graceful shutdown.

### `proto/<package>/config/v1/config.proto`

Protobuf definition of the application's configuration schema. Defines five
message types:

**`Config`** (root):

| Field | Type | Description |
|-------|------|-------------|
| `app` | `AppConfig` | Server-side settings |
| `public` | `PublicConfig` | Browser-safe settings served to the frontend |
| `database` | `DatabaseConfig` | PostgreSQL connection and pool settings |

**`DatabaseConfig`**:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `dsn` | `string` | `postgres://...localhost:5432/<app>` | Postgres connection string (secret — not exposed via flags) |
| `max_conns` | `int32` | `0` (pgxpool default) | Maximum pool connections |
| `min_conns` | `int32` | `0` (pgxpool default) | Minimum idle connections |
| `max_conn_lifetime` | `string` | `""` (pgxpool default: 1h) | Go duration string for max connection reuse time |
| `max_conn_idle_time` | `string` | `""` (pgxpool default: 30m) | Go duration string for max idle time |
| `health_check_period` | `string` | `""` (pgxpool default: 1m) | Go duration string for health check interval |
| `auto_migrate` | `bool` | `false` | Run goose migrations on startup |

Duration fields use Go duration string format (e.g., `1h`, `30m`, `1m`).
Empty strings leave the pgxpool defaults in place. The `dsn` field is marked
as secret and is not exposed as a CLI flag.

**`AppConfig`**:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `string` | App directory name | Application name for logs |
| `port` | `int32` | `3000` | HTTP server port |

**`PublicConfig`**:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `app_name` | `string` | App directory name | Display name in the browser |
| `api_base_url` | `string` | `"http://localhost:3000"` | Base URL for API calls |
| `auth` | `PublicAuthConfig` | — | OIDC authentication settings |

**`PublicAuthConfig`**:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `issuer` | `string` | `"http://localhost:9000"` | OIDC issuer URL |
| `client_id` | `string` | `"<app>-web"` | OIDC browser client ID |
| `scopes` | `repeated string` | `["openid", "profile", "email"]` | OIDC scopes |
| `redirect_path` | `string` | `"/auth/callback"` | Auth callback path |
| `post_logout_redirect_path` | `string` | `"/"` | Post-logout redirect |

All defaults use `gofra.config.v1.field` annotations. Run
`gofra generate config` after editing to regenerate the Go code.

### `gofra.yaml`

Default configuration overrides. Proto defaults apply when a key is omitted.

```yaml
app:
  name: <app-name>
  port: 3000

database:
  dsn: "postgres://postgres:postgres@localhost:5432/<app-name>?sslmode=disable"
  auto_migrate: true
```

This file is read by `runtimeconfig.Load` as the YAML layer. It can be
overridden by environment variables or CLI flags.

### `mise.toml`

Task runner definitions:

| Task | Depends On | Description |
|------|-----------|-------------|
| `generate` | — | Generates config code from proto via `gofra generate config` |
| `gen:sql` | — | Generates Go code from SQL queries via `sqlc generate` |
| `dev` | `generate` | Generates code then runs `go run ./cmd/app` |
| `migrate` | — | Runs pending database migrations via goose |
| `migrate:create` | — | Creates a new migration file |
| `migrate:down` | — | Rolls back the last migration |
| `migrate:status` | — | Shows migration status |
| `seed` | — | Seeds the database with development data |

The `generate` task uses `GOFLAGS=-mod=mod` to allow the local framework
replace directive to work with `go run`. The `migrate`, `migrate:*`, and
`seed` tasks require the `DATABASE_URL` environment variable.

### `db/`

- **`embed.go`** — Uses `//go:embed migrations/*.sql` to bundle migration
  files into the binary. Exports `var Migrations embed.FS` for use with
  `runtimedatabase.Migrate`.
- **`migrations/`** — goose SQL migration files with `-- +goose Up` /
  `-- +goose Down` annotations. File names are zero-padded
  (`00001_create_posts.sql`) so sqlc parses them in order.
- **`queries/`** — sqlc query files. Each `.sql` file contains named queries
  (`-- name: GetPost :one`) that sqlc compiles into type-safe Go code.
- **`seeds/`** — Seed data applied via `goose --no-versioning`. Written to
  be idempotent (`ON CONFLICT DO NOTHING`).
- **`sqlc/`** — Generated by `mise run gen:sql`. Do not edit.

### `sqlc.yaml`

sqlc configuration. Points `schema` at `db/migrations/` (sqlc reads goose
migration files as the database schema), `queries` at `db/queries/`, and
generates Go code into `db/sqlc/` using `pgx/v5`.

### `web/`

- **`index.html`** — SPA starter page. Loads `/_gofra/config.js` via a
  `<script>` tag and accesses the config via `window.__GOFRA_CONFIG__`.
  Displays the app name, API base URL, and full config JSON. Uses camelCase
  property names (`runtimeConfig.appName`, `runtimeConfig.apiBaseUrl`).
- **`embed.go`** — Uses `//go:embed index.html` to bundle assets into the
  binary. Exports a `Handler()` function that returns an
  `http.FileServerFS` handler.

### `go.mod`

The generated `go.mod` includes a local `replace` directive pointing to the
framework checkout that created the app:

```
replace databit.com.br/gofra => /path/to/gofra
```

This is necessary because the framework is not yet published. Once published,
this replace directive can be removed.

## Runtime Behavior

On startup, the app connects to PostgreSQL using `database.dsn`, optionally
runs embedded migrations when `database.auto_migrate` is true, and registers
a Postgres readiness check. If the database is unreachable or migrations
fail, the app exits immediately (fail-fast).

The generated app listens on the port configured in `gofra.yaml` (field
`app.port`, default `3000`). It exposes:

| Path | Handler | Source | Purpose |
|------|---------|--------|---------|
| `/startupz` | Root mux | `runtimehealth` | Startup probe — returns 200 after init |
| `/livez` | Root mux | `runtimehealth` | Liveness probe — always returns 200 |
| `/readyz` | Root mux | `runtimehealth` | Readiness probe — 200 if checks pass |
| `/_gofra/config.js` | App router | Generated `PublicConfigHandler` | Public config as JavaScript |
| `/*` | App router | `web.Handler()` | Embedded SPA assets |

Health probes are on the root mux, structurally outside the app router's
middleware chain. This ensures health checks work even if app middleware
fails.

The config handler serves the `PublicConfig` subtree only — `AppConfig`
fields (like `port`) and any secret fields are never sent to the browser.

Graceful shutdown is handled by `runtimeserve.Serve()` with the health
checker wired as the `Health` provider. See
[runtime/serve](../runtime/serve.md) for shutdown sequence details.
